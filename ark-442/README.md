# ARK-442 — Authorization Boundary Degradation Under Verification-to-Execution Delay

**Remnant Fieldworks Inc. — Derek Hone**
**Backend:** IBM Quantum `ibm_marrakesh` (preferred) or `ibm_fez` (Heron r2) · **Qubits:** selected at execution time (RE<2%, connected, min-sum) · **Instance:** open-instance
**Governing principle:** *Proof Before Power. Prediction Before Measurement. No Rescue After Failure.*

ARK-442 is a preregistered metrological characterization of how the verify-then-execute (VBE) authorization boundary degrades as the separation between authorization measurement and payload execution increases. It is **not new physics and makes no cryptographic claim** — the boundary erosion it measures is decoherence-driven (T1/T2 decay of the authorization qubit Q_A between measurement and payload execution). Eight arms probe: ALLOW survival at four delays (0 / 0.5 / 1.0 / 2.0 µs), an expired authorization that must DENY, a replayed stale classical bit that must not authorize, a fresh reverification that must restore ALLOW, and an idle SPAM baseline. Locked thresholds carry over from ARK-441: the DENY arms (expired, replayed) must satisfy `L_D_corrected ≤ 0.02` and `Δ_B(Arm 1 vs Arm 5) ≥ 0.70`; reverification must restore `S_reverified ≥ 0.90`; the delay-decay curve `S_A(delay)` is reported descriptively (not pass/fail). The in-situ SPAM job runs first and gates the principal job — if `SPAM_baseline > 2%` the experiment is KILLED (INDETERMINATE) and stops immediately with no rescue. The full 28-field preregistration (`ARK_442_preregistration.md`) and all code are committed with SHA-256 hashes **before any IBM Quantum job is submitted**; the preregistration commit hash is the lock, recorded in `RUN_LOG.md`.
