# ARK-449 — State Changes After Verification
**ExecutionProof ARK Authorization-Boundary Track**  
**Remnant Fieldworks Inc.**  
**Status:** PRE-EXECUTION — awaiting IBM budget refresh and lock

## One-Line Summary

Verifies that the ExecutionProof control model denies execution when world state
has changed between authorization (T₁) and execution (T₃) — and allows execution
only when re-verified against currently admissible state.

**Doctrine tested:**
> Permission at approval time is not permission at execution time.

---

## Why This Experiment Matters

Every prior ARK experiment tested a static boundary: given a fixed world, does the
system enforce authorization correctly? ARK-449 tests the boundary under movement:
the world state changes between when an agent received approval and when it acts.

This is the dominant failure mode in real enterprise AI deployments. An agent
approved to transfer funds when the account had sufficient balance, approved to
access a system when the credential was valid, approved to contact an entity before
it was sanctioned — these are not edge cases. They are the normal operating
condition of any asynchronous agentic system.

The result either demonstrates that ExecutionProof's state-check control logic
survives this condition, or it precisely identifies which class of state change
the current implementation fails to contain.

## Experiment Design

**Qubits:** Q_A (authorizer), Q_P (payload) — 2 qubits, classical feedforward only  
**Arms:** 9, submitted in a single principal job  
**Shots:** 8192 per arm (principal), 2048 (SPAM gate)  
**Backend:** ibm_marrakesh (156-qubit Heron r2)

| Arm | Label | c_state | Scenario |
|---|---|---|---|
| 1 | ALLOW-unchanged | 1 | State valid at T₁ and remains valid at T₃ |
| 2 | DENY-revoked | 0 | Authority revoked between approval and execution |
| 3 | DENY-policy | 0 | Policy updated; action now inadmissible |
| 4 | DENY-balance | 0 | Account balance fell below threshold after approval |
| 5 | DENY-risk | 0 | Risk score crossed enforcement limit after approval |
| 6 | DENY-destination | 0 | Destination blocked/sanctioned after approval |
| 7 | DENY-expiry | 0 | Supporting evidence expired after approval |
| 8 | ALLOW-reauth | 1 | Fresh authorization issued under re-assessed state |
| 9 | DENY-replay | 0 | Old ProofRecord presented; state has changed |

**Pass criteria (all four required simultaneously):**

| Criterion | Threshold |
|---|---|
| C1: S_A_min (minimum ALLOW retention) | ≥ 0.90 |
| C2: L_D_max (maximum DENY leakage, any arm) | ≤ 0.02 |
| C3: Δ_B = S_A_min − L_D_max | ≥ 0.70 |
| C4: SPAM gate (SPAM_A and SPAM_P) | ≤ 0.02 each |

## Execution Protocol (strict ordering required)

```
Step 1  Compute SHA-256 hashes of preregistration + circuit files
        → fill MANIFEST.txt → commit → push → tag ark-449-v1.0-lock
        *** NO CHANGES AFTER THIS POINT ***

Step 2  Run qubit selection (built into ark_449_circuit.py)
        → selected_qubits.json written
        → commit before submitting SPAM gate

Step 3  Submit SPAM gate job
        → spam job ID written to execution_log.json
        → commit before reading any results

Step 4  Submit principal job (9 arms)
        → principal job ID written to execution_log.json
        → commit before reading SPAM results

Step 5  Read SPAM gate results → evaluate gate
        If SPAM gate FAILED:
            → record abort → stop → tag ark-449-v1.0 → merge PR
        If SPAM gate PASSED:
            → continue to Step 6

Step 6  Read principal results
        → raw_results.json written

Step 7  Run ark_449_analysis.py
        → analysis_results.json written
        → proofrecord.json written
        → verdict printed to stdout

Step 8  Commit all results → push → tag ark-449-v1.0 → open PR → merge
        → update Zenodo
```

**Lock rules (immutable after Step 1):**
- No criterion changes, loosening, or additions after MANIFEST commit
- No arm additions or removals
- No rescue-after-failure if SPAM gate fails
- No post-hoc subgroup selection
- SPAM_P is a gate diagnostic only — never subtracted from L_D
- Qubit pair fixed at SPAM gate submission
